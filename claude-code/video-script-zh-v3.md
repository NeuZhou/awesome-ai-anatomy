## 给NotebookLM的输入文档

以下是Claude Code架构拆解的视频讲解稿。请生成一期播客式对话讲解。

---

世界上最先进的AI编程助手，内部居然藏了一个虚拟宠物系统。

18种宠物。5个稀有度等级。还有RPG属性——调试力、耐心值、混乱值、智慧值、毒舌值。1%的概率出闪光变体。

我没开玩笑。这是Anthropic的Claude Code。51万行源码，上周意外泄露了。

不是被黑客攻破的。有人发现npm包里残留了一个.map文件，里面有个链接指向Anthropic的云存储。那个链接没有任何访问控制。点进去，1903个文件的完整TypeScript源码，直接下载。

我花了好几天时间把它拆了一遍。宠物系统只是开胃菜。真正让我震惊的，是这个AI Agent的内部架构。

下面我分三个级别来说。

第一级：聪明，但你可能猜得到。

Claude Code的终端界面，是用React写的。

刚看到的时候我也觉得离谱——一个命令行工具，UI层用React？但想想就明白了。这个工具要同时处理好几个Agent并行跑、流式输出、用户随时打断、权限弹窗……这种状态管理的复杂度，用传统的终端库写出来的代码三个月后自己都看不懂。React的声明式模型反而是最合理的选择。

另一个不意外但值得说的：整个Agent的核心是一个while真循环。一个文件，query.ts，1729行。循环逻辑就是：整理上下文，调Claude的API，检测输出里有没有工具调用，有就执行，执行完把结果塞回去继续循环。没有就退出。

听起来简单对吧？魔鬼在第二级。

第二级：真正让我大吃一惊的东西。

首先是上下文管理。你跟Claude Code聊久了，对话会变得很长。大部分AI工具怎么处理？简单粗暴——砍掉前面的对话。

Claude Code不是。它有四层压缩机制，我管它叫四把手术刀。

打个比方。你手机满了要清空间。第一步你删不用的APP。第二步你清缓存。第三步你把老照片压缩存云端。第四步实在不行了，恢复出厂设置。

Claude Code也是这样。第一层是精准删除——把用完的工具调用结果直接删掉，不做任何摘要。第二层是在API的缓存层面做手术——消息内容不变，但告诉Claude那些Token别看了。第三层是把旧对话归档成结构化的摘要，像git的提交日志。第四层才是最粗暴的全量压缩。

这四层按顺序触发。如果前两层就够了，后面的根本不启动。

为什么这很重要？因为这体现了一个设计原则：先无损再有损，先局部再全局。大部分AI工具只有第四层。Claude Code有四层。这就是为什么它处理长对话比别人好。

然后是流式工具并行执行。这个是我觉得整个代码库里最精巧的设计。

普通的AI Agent是怎么干的？等模型说完话，检查有没有工具调用，有的话执行，执行完再下一轮。Claude Code不等。模型还在说着话呢，检测到一个工具调用就立刻开始执行。读文件、搜索这种只读操作同时跑，写文件这种需要排队。

这就是为什么Claude Code感觉比很多竞品快。不是模型快——是工程优化把等待时间都砍了。

第三级：完全没想到的东西。

除了开头说的宠物系统，代码里还有一整套语音模式，内部代号叫琥珀石英。有一个远程控制模式——你运行一个命令，你的电脑就变成claude网页版的远程终端。还有消融实验基础设施——就是可以系统性地关掉某个功能，然后量化它到底有多少贡献。

这最后一个最让我佩服。大部分公司做A/B测试，但Claude Code团队做的是消融实验。这是机器学习论文里的标准方法论——逐个关掉组件看性能变化——但在产品工程里几乎没人这么做。

这说明这个团队不是在做产品，是在用做研究的方法做产品。每一个功能，都有数据支撑它存在的理由。

对了，宠物系统除外。那个纯粹是因为好玩。

插一个有意思的细节。所有18个宠物的名字在源码里都用十六进制编码了。为什么？因为其中一个宠物名和Anthropic某个还没发布的模型内部代号撞了。构建系统会扫描输出文件有没有被排除的字符串，所以全部编码。

至于是哪个名字——duck、goose、cat、dragon、octopus、owl、penguin、turtle、snail、ghost、axolotl、capybara、cactus、robot、rabbit、mushroom、chonk——18个里面有一个是下一个模型的代号。到现在也没人确认到底是哪个。

谈一个有争议的观点。

读完全部代码之后，我有一个不太受欢迎的判断：Claude Code有过度工程的倾向。

四层上下文管理。40多个工具。双层Feature Flag系统。消融实验基础设施。虚拟宠物。1729行的单文件核心循环。

这对一个研究导向的团队来说是合理的——他们需要测量一切。但对两年后要维护这个代码库的人来说，这可能是噩梦。

最后说说泄露本身怎么发生的。

npm发布的时候忘删.map文件。map文件里有个R2存储桶的链接。链接没有访问控制。就这么简单。

51万行核心产品源码，被一个忘删的文件暴露了。

给所有发npm包的人提个醒：package.json的files字段用白名单制。CI里加一步检查发布产物有没有.map文件。源码归档的URL一定要加鉴权。

总结

读完这51万行代码，你让我说一个最大的收获——

不是某个具体技术。是这个团队对待工程的态度。

他们用做论文的方法做产品。每个功能都有量化验证。每个设计决策都有明确的trade-off分析。这种"研究型工程文化"值得每一个做AI产品的团队学习。
