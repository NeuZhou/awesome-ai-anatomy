# Claude Code 拆解 — 中文视频脚本（NotebookLM用）

## 给NotebookLM的输入文档

以下是Claude Code的架构拆解分析，请根据这份材料生成一期播客式讲解视频。

---

你知道吗？上周发生了一件AI圈的大事——Anthropic的Claude Code，51万行源码，全泄露了。

不是被黑客攻破的。是有人发现npm包里留了个.map文件，顺藤摸瓜找到了一个没加密的下载链接。点进去就是完整的未混淆TypeScript源码。1903个文件，51万行，摊在眼前。

这大概是我们第一次能完整看到一个顶级AI Agent的工业级源码。不是demo，不是教程，是真正在生产环境跑的、服务了几百万开发者的Agent代码。

我花了不少时间把它拆解了一遍。下面说说最有意思的发现。

第一个让我意外的事：这个终端工具的界面，是用React写的。

对，你没听错。一个命令行工具，UI层用的是React加Ink框架。你看到的那些权限弹窗、进度条、工具调用的实时展示，底下全是JSX组件。

一开始觉得过度设计了。但想想也合理——多个Agent并行跑、流式输出、用户随时可能中断，这种复杂状态管理用React的声明式模型确实比手搓靠谱。

第二个发现，也是我觉得最精妙的：整个Agent的核心就是一个while true循环。

真的就一个文件，query.ts，1729行。逻辑很直白：预处理上下文、调Claude API、一边接收流式输出一边检测有没有工具调用、有的话执行完把结果塞回去继续循环、没有的话退出。

看着简单对吧？但魔鬼在细节里。

比如上下文管理。大部分Agent怎么处理长对话？对话太长了就摘要一下，砍掉前面的。Claude Code不是这样。它有四层压缩机制，像四把手术刀：

第一层叫History Snip，最精细的，直接把特定的消息范围删掉，不做任何摘要。比如一个工具调用的中间结果，用完了就直接删，不浪费token。

第二层叫Microcompact，这个比较黑科技。它不改消息内容，而是在API的缓存层面做编辑——告诉Claude"这些token你缓存里有，但别用了"。

第三层叫Context Collapse，把旧的对话轮次归档成结构化摘要，类似git的提交日志。不是一坨摘要文本，是有结构的归档。

第四层才是Autocompact，最粗暴的全量压缩，最后的兜底手段。

这四层按顺序执行。如果前两层已经把上下文压到阈值以下了，后面的就不触发。

这给我一个很大的启发：做Agent的上下文管理，不能只有一种策略。不同的信息"过期速度"不一样——工具调用结果过期很快，用户的偏好设定过期很慢。得分层处理。

还有一个设计让我觉得非常聪明：流式工具并行。

普通Agent的流程是：等模型说完、看有没有工具调用、有的话执行、结果返回、下一轮。Claude Code不等。模型还在流式输出后面的内容，前面检测到的工具调用已经开始执行了。

读文件、搜索这种只读操作可以同时跑，写文件这种需要独占锁。结果按接收顺序缓冲，不会乱序。

这就是为什么Claude Code用起来感觉比很多竞品快。不是模型快，是工程优化省掉了大量等待时间。

工具系统的设计也值得说。40多个工具，没有一个class继承，全是纯函数式的工厂函数。每个工具完全自包含——输入校验、权限检查、执行逻辑、终端UI渲染、上下文摘要，全在一个文件里。

其中最复杂的是BashTool。它不是简单地exec一个命令：会自动分类命令类型，macOS上走沙箱执行，超过15秒的命令自动转后台，大输出存到磁盘然后只给模型一个文件路径。

还有一些有趣的彩蛋。代码里藏了一个完整的虚拟宠物系统——18种宠物，5级稀有度，还有RPG属性：调试力、耐心值、混乱值、智慧值、毒舌值。1%的概率出闪光变体。

所有宠物的名字都用十六进制编码了。为什么？因为有一个宠物名和Anthropic某个未发布模型的内部代号撞了。构建系统会检测输出文件有没有被排除的字符串，所以全编码避免误触发。

那到底是哪个名字呢？duck、goose、cat、dragon、octopus、owl、penguin……18个里面有一个是下一个模型的代号。有兴趣的可以猜猜。

最后说一下这次泄露的原因。技术上极其简单：npm发布时忘删.map文件，map里有个R2存储桶的链接，链接没有访问控制。就这么简单。

给所有发npm包的人提个醒：package.json的files字段用白名单，CI里检查发布产物有没有.map文件，源码归档的URL一定要加鉴权。

读完51万行代码，我最大的感受不是某个具体技术多厉害。而是这个团队在用做研究的方法做工程——消融实验基础设施、双层feature flag、四种粒度的上下文管理、流式工具并行。每个功能背后大概率都有数据验证。

这种"每个特性都有量化验证"的工程文化，比任何单点技术都更值得学。

当然，宠物系统除外。那个纯粹是因为好玩。
