# Contributing to Awesome AI Anatomy

Thanks for wanting to contribute! Here's how.

## 🐛 Fix Errors
Found a factual error in a teardown? Open a PR with:
- The specific claim that's wrong
- Evidence of the correct information
- Link to source code or documentation

## 🔍 Enhance Existing Teardowns
- Add missing architecture details
- Improve diagrams
- Add benchmark data
- Translate to another language

## 💡 Suggest a Project to Teardown
Open an issue with the `project-suggestion` label:
- Project name and repo URL
- Why it's interesting architecturally
- Approximate lines of code
- What questions you'd want answered

## ✍️ Write a New Teardown
Want to do a full teardown? Amazing. Here's how:

1. Fork the repo
2. Create `project-name/README.md` using [TEMPLATE.md](TEMPLATE.md)
3. Include:
   - At a Glance table with verified stats
   - Architecture overview with Mermaid diagram(s)
   - Core innovation analysis with code references
   - The Verdict (mixed positives/negatives, not separate lists)
   - Cross-project comparison table
   - Verification log (all numbers fact-checked)
4. Run through the quality checklist in TEMPLATE.md
5. Submit PR

## 🌍 Translate
Help translate existing teardowns. Create `README.{lang}.md` in the project folder.

## Quality Standards
- All claims must be verifiable from source code
- All numbers (stars, LOC, dates) must be fact-checked
- Include file paths and line numbers for code references
- Opinions must be clearly stated as opinions
- Architecture diagrams must use Mermaid (GitHub-renderable)
- No AI-generated filler text — write like a real engineer

## Code of Conduct
Be respectful of the projects you're analyzing. Teardowns are critical analysis, not attacks. Point out problems with evidence, not snark.
